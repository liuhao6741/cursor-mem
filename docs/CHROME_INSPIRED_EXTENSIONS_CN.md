# 以 Chrome 演进为参照的 cursor-mem 扩展方向

本文档以 Chrome 浏览器的功能演进为参考，映射到 Cursor 使用场景，提出 cursor-mem 可扩展的功能方向，用于解决用户使用 Cursor 时的常见痛点。

---

## 一、Chrome 演进与 Cursor 痛点映射

| Chrome 能力 | 解决的用户问题 | 在 Cursor / cursor-mem 中的对应 |
|-------------|----------------|--------------------------------|
| **历史 + 搜索** | “上次看的页面在哪？” | 已有：FTS5 + MCP 检索；可增强：统一入口、建议 |
| **书签** | “这个页面要常来” | 缺失：重要会话/观察「钉选」，始终进上下文 |
| **重新打开已关闭标签** | “手滑关了，要回来” | 缺失：「从上次会话继续」一键恢复上下文 |
| **多配置文件** | 工作/个人分离 | 缺失：按「场景」的记忆策略（如 code-review / 写文档） |
| **无痕模式** | “这次别记” | 缺失：本会话不写入 cursor-mem |
| **同步** | 多设备一致体验 | 缺失：同一项目在笔记本/台式机共享记忆（可选、加密） |
| **扩展/插件** | 自定义行为 | 缺失：自定义压缩器、上下文策略、MCP 工具 |
| **地址栏建议** | 边输入边联想 | 缺失：输入 prompt 时基于历史的建议（“你常做的下一步”） |
| **侧边栏** | 书签/历史常驻 | 部分：Web 查看器；可增强：规则内「当前上下文」可观测 |
| **性能/资源** | 不卡、省资源 | 可增强：老会话「休眠」、按需加载、token 预算可视化 |

下面按**用户痛点**列出可扩展功能，并标注与 Chrome 的类比及在路线图中的位置。

---

## 二、可扩展功能清单（按痛点）

### 1. “上次做到哪了？”——继续上次会话

**Chrome 类比**：重新打开已关闭的标签 / “从上次中断的地方继续”。

**痛点**：新开 Chat 后，希望 Agent 立刻从「上次未完成的那一步」接着做，而不是只看到摘要。

**扩展方向**：

- **「恢复上次会话」**  
  - 在规则或 Web 中提供一键「注入上次会话的最后 N 条观察 + 未完成任务描述」。  
  - 新 Chat 打开时可选：“从上次会话继续” → 自动带上前次结尾的 prompt/文件/命令。  
- **「未完成」标记**  
  - 会话若在 stop 前被用户主动结束，可标记为 `incomplete`，下次优先提示“是否继续？”。

**与 ROADMAP 关系**：属于「上下文策略」的增强（计划中·中期）。

---

### 2. “这段对话别记”——隐私 / 无痕

**Chrome 类比**：无痕模式（Incognito）。

**痛点**：调试敏感信息、试跑危险命令、或单纯不想让某次对话污染记忆时，希望「本会话不写入 cursor-mem」。

**扩展方向**：

- **会话级“不记录”**  
  - 在 Cursor 中通过某种方式标记当前 Chat（如规则里的标签、或未来 Cursor 提供的 hook 参数）为 `no_memory`。  
  - hook 收到后：本会话的 observations 不写入 DB，或写入临时表并在会话结束时丢弃。  
- **范围控制**  
  - 配置项：仅记录某目录下编辑、或排除某些路径/命令（如 `rm -rf`、带 token 的命令），类似扩展的权限粒度。

**与 ROADMAP 关系**：与「数据与隐私」中的脱敏/排除一致（计划中·中期）；无痕可单独作为小 feature。

---

### 3. “这个会话/结论要常看”——书签与钉选

**Chrome 类比**：书签（Bookmarks）。

**痛点**：某次会话的结论（如“我们决定 API 用 REST 不用 GraphQL”）或某条观察需要长期保留在上下文中，而不是被时间线冲掉。

**扩展方向**：

- **钉选观察 / 钉选会话**  
  - 用户或 Agent 可将某条 observation 或整场 session 标记为 `pinned`。  
  - 构建上下文时：pinned 内容优先占用 budget，或单独一个「常驻片段」区块。  
- **“重要会话”标签**  
  - 与 ROADMAP 中「按会话标签或手动标记重要会话优先注入」一致；可提供“标记为重要”的入口（Web + 后续 Cursor 集成）。

**与 ROADMAP 关系**：计划中·中期「上下文策略」已包含“重要会话”优先注入。

---

### 4. “工作/写文档/Code Review 分开记”——多配置/场景

**Chrome 类比**：多配置文件（Profiles）。

**痛点**：同一项目下有时在写代码、有时在 Code Review、有时在写文档，希望不同「场景」下注入的记忆侧重点不同（例如 review 时更关注最近改动的文件，写文档时更关注设计决策）。

**扩展方向**：

- **记忆场景（Profile）**  
  - 配置多种场景：如 `default`、`code-review`、`documentation`。  
  - 每个场景可配置：context_budget 分配、是否包含最近 N 条 shell、是否优先注入「钉选」或「带某 tag」的会话。  
- **按场景选择**  
  - 通过规则内的标记、或 Cursor 项目/工作区设置，选择当前 Chat 使用的场景，builder 按该场景组装上下文。

**与 ROADMAP 关系**：与「上下文策略」中的可配置策略一致，可细化为「场景」维度。

---

### 5. “家里和公司都要接着干”——同步与多设备

**Chrome 类比**：Chrome Sync（书签、历史、设置跨设备）。

**痛点**：同一仓库在办公室和家里都有 clone，希望两边的 Cursor 都能看到同一套「项目记忆」，而不是各记各的。

**扩展方向**：

- **可选同步**  
  - 将 `~/.cursor-mem`（或指定 DATA_DIR）同步到用户自己的存储（如 iCloud、Dropbox、自建 NAS），或提供「导出/导入」包，在另一台机器上导入后继续。  
- **端到端加密**  
  - 若同步到第三方存储，支持本地加密再上传，密钥仅在本机，避免明文记忆泄露。  

**与 ROADMAP 关系**：长期·探索「协作与同步」中已提及；先做导出/导入增强更稳妥，再考虑真正的多端同步。

---

### 6. “我输入时给我提示”——智能建议

**Chrome 类比**：地址栏输入时的历史与搜索建议（Omnibox）。

**痛点**：在输入 prompt 时，希望基于历史得到建议，例如“你上次在类似情况下接着做了 X”“经常在改完 A 之后跑 B 命令”。

**扩展方向**：

- **Prompt 建议 API 或 MCP**  
  - 根据当前输入（或当前打开文件）在历史中做 FTS 或向量检索，返回「相似历史 prompt」或「常做的下一步」简短列表。  
  - 可作为 MCP 工具（如 `memory_suggest`）供 Agent 调用，或未来由 Cursor 在输入框旁展示（需 Cursor 支持）。  
- **“接下来可能要做”**  
  - 在规则注入的摘要末尾加一句：根据最近会话推断的「下一步建议」（基于规则或简单统计），减少用户重复描述。

**与 ROADMAP 关系**：检索增强（向量检索、更细粒度过滤）可支撑“相似 prompt”；建议层可放在中期。

---

### 7. “记忆里到底塞了啥？”——可观测性与调试

**Chrome 类比**：DevTools（检查当前页面状态与性能）。

**痛点**：用户或开发者想确认「当前这次对话，到底注入了哪些会话、哪些观察、用了多少 token」，以便调试规则或 budget。

**扩展方向**：

- **“当前上下文”预览**  
  - CLI 或 Web：`cursor-mem context preview --project xxx` 或页面中展示「本次若现在发 prompt，会注入的完整内容」的只读预览。  
- **Token 预算分解**  
  - 在 builder 中按段（摘要 / 最近操作 / 关键文件 / 钉选）统计字符或估算 token，在 status 或 Web 中展示，便于调 context_budget 和策略。  
- **为什么这条被选中**  
  - 对某条出现在上下文中的观察，可标记来源（例如“来自最近会话摘要”“来自钉选”“来自最近操作”），便于理解与排错。

**与 ROADMAP 关系**：可观测性在短期「配置与可观测性」中已有；可细化为「上下文预览 + token 分解」。

---

### 8. “历史太多，别卡也别爆”——性能与规模

**Chrome 类比**：标签页休眠、懒加载、历史归档。

**痛点**：项目用了很久，会话和观察数量很大，不希望每次构建上下文或搜索都变慢，也不想无上限占用磁盘。

**扩展方向**：

- **老会话“休眠”**  
  - 超过 N 天未参与的会话：只保留摘要 + 关键元数据，观察详情不加载；仅在 MCP `memory_get(ids=[...])` 时按需从冷存储或压缩块读取。  
- **观察采样/归档**  
  - 与 ROADMAP 一致：对非常老的观察做采样（例如只保留 10% 的 shell 输出）或归档到单独表/文件，FTS 只索引摘要与标题。  
- **分层存储**  
  - 热数据（最近 7 天）：全量；温数据（7–30 天）：仅摘要 + 标题；冷数据：仅摘要，详情按需加载。

**与 ROADMAP 关系**：长期「性能与规模」已包含索引优化、采样与归档；可细化分层策略。

---

### 9. “按主题/任务看历史”——分组与集合

**Chrome 类比**：标签组、收藏夹/集合。

**痛点**：想按「某次重构」「某次修 bug」「某次写文档」来看历史，而不是纯时间线。

**扩展方向**：

- **会话标签/主题**  
  - 通过摘要或可选 AI 给会话打标签（如 `refactor`、`bugfix`、`docs`）；用户或 Agent 可按标签过滤搜索与时间线。  
- **集合（Collection）**  
  - 用户手动或通过规则将多场会话归为「集合」（如“v0.2 发布相关”），上下文注入时可选择「优先包含某集合的摘要」。  

**与 ROADMAP 关系**：「按会话标签或手动标记重要会话」已列入中期；集合可视为标签的批量视图。

---

### 10. “一个入口搜一切”——统一记忆入口

**Chrome 类比**：Omnibox（一个框：URL + 搜索 + 历史）。

**痛点**：想在一个地方查「过去改过的文件、跑过的命令、某次对话的结论」，而不是分别用 Cursor 历史 + 本机 shell 历史 + 自己回忆。

**扩展方向**：

- **统一记忆搜索**  
  - Web 或未来 Cursor 侧边栏：一个搜索框，同时搜 observations + sessions，支持类型过滤、日期范围、项目，结果统一展示（时间线 + 类型图标）。  
  - MCP 已提供 memory_search；这里强调的是「单一入口 + 更好展示」。  
- **与 Cursor 历史联动（若 Cursor 开放 API）**  
  - 若 Cursor 提供历史查询 API，可在同一入口合并「Cursor 原生对话标题」与「cursor-mem 的结构化观察」，实现“一次搜索，两种历史”。

**与 ROADMAP 关系**：Web 查看器已有搜索；增强为「默认首页即搜索 + 筛选」即可向 Omnibox 体验靠拢。

## 三、优先级建议（结合 Chrome 价值与实现成本）

| 优先级 | 功能 | 痛点强度 | 实现成本 | Chrome 类比 |
|--------|------|----------|----------|-------------|
| 高 | 恢复上次会话 / 从上次继续 | 高 | 中 | 重新打开已关闭标签 |
| 高 | 钉选观察/会话（书签） | 高 | 中 | 书签 |
| 中 | 本会话不记录（无痕） | 中 | 低 | 无痕模式 |
| 中 | 当前上下文预览 + token 分解 | 中 | 低 | DevTools |
| 中 | 记忆场景（Profile） | 中 | 中 | 多配置文件 |
| 中 | Prompt/下一步建议 | 中 | 中 | 地址栏建议 |
| 低 | 导出/导入增强，可选同步 | 中 | 中高 | Sync |
| 低 | 会话标签与集合 | 中 | 中 | 标签组/集合 |
| 低 | 老会话休眠与分层存储 | 低（当前） | 中 | 标签休眠/归档 |

---

## 四、小结

以 Chrome 为参照，cursor-mem 可以重点在以下几类能力上演进：

1. **“从上次继续”**：恢复上次会话、未完成标记，减少重复说明。  
2. **“有的要常留、有的别记”**：书签/钉选 + 无痕/范围控制，兼顾重要性与隐私。  
3. **“看得见、可调试”**：上下文预览与 token 分解，提升可观测性。  
4. **“按场景与主题用记忆”**：多场景配置、标签与集合，更贴合不同使用方式。  
5. **“一个入口搜一切”**：统一记忆搜索与展示，接近 Chrome Omnibox 的体验。  
6. **“长期不卡、可同步”**：休眠/分层/归档 + 可选同步，支撑长期使用与多设备。

上述方向与现有 ROADMAP 中的「上下文策略」「数据与隐私」「检索增强」「性能与规模」「协作与同步」均可对应；本文可作为按「用户痛点 + Chrome 类比」做需求拆解和迭代的参考。
